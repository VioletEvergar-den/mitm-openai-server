# mitm-openai-server 后端代码架构

## 整体架构

mitm-openai-server 采用模块化设计，主要包含以下核心组件：

- **cmd**: 命令行入口及参数解析
- **pkg**: 核心功能模块
  - **api**: API处理层
  - **openai**: OpenAI API模拟和代理实现
  - **server**: HTTP服务器实现
  - **storage**: 数据存储层
  - **utils**: 工具函数
  - **embed**: 静态资源嵌入

## 核心模块详解

### cmd 模块

负责命令行参数解析和服务启动，使用cobra库实现命令行功能。

- `server.go`: 实现服务器启动命令，处理命令行参数并初始化服务器
- `root.go`: 定义根命令和全局标志

### pkg/openai 模块

实现OpenAI API的模拟和代理功能，采用接口设计模式。

- `interface.go`: 定义Service接口，统一模拟服务和代理服务的API
- `service.go`: 实现服务工厂，根据配置创建模拟或代理服务
- `mock.go`: 实现模拟OpenAI API服务，返回虚拟响应
- `proxy.go`: 实现代理服务，将请求转发至真实API
- `config.go`: 定义服务配置结构
- `handlers.go`: 实现API请求处理和记录

### pkg/server 模块

实现HTTP服务器，管理路由和中间件。

- `server.go`: 核心服务器实现，包含路由设置和中间件
- `config.go`: 服务器配置管理
- `models.go`: 服务器相关数据模型

### pkg/storage 模块

实现数据存储层，支持文件系统和SQLite存储。

- `storage.go`: 定义Storage接口
- `models.go`: 定义请求和响应数据模型
- `filesystem.go`: 基于文件系统的存储实现
- `sqlite.go`: 基于SQLite数据库的存储实现

### pkg/api 模块

实现API处理功能，包含多个子模块：

- `models.go`: 定义API相关数据结构
- `openai_handlers.go`: OpenAI API请求处理
- `requests_handlers.go`: 请求记录查询API
- `ui_handlers.go`: UI界面后端API

## 关键流程

### 请求处理流程

1. 客户端发送请求至服务器
2. 服务器中间件记录请求基本信息
3. 根据工作模式路由请求：
   - 独立模式：由mock服务处理并返回模拟响应
   - 代理模式：由proxy服务转发至目标API并返回真实响应
4. 中间件记录完整请求与响应信息
5. 保存请求记录至存储层

### 代理模式配置流程

1. 通过命令行参数或UI界面设置代理模式
2. ConfigManager加载/保存配置
3. 配置应用至服务器
4. 服务器根据配置创建对应的OpenAI服务实现

## 设计模式

- **工厂模式**: 通过ServiceCreator创建不同的服务实现
- **策略模式**: 通过Service接口支持不同的OpenAI API实现策略
- **依赖注入**: 各组件通过构造函数注入依赖
- **中间件模式**: 使用Gin框架的中间件处理横切关注点
- **仓储模式**: 通过Storage接口抽象数据存储

## 扩展性设计

系统设计考虑了多方面的扩展性：

1. **存储扩展**: 通过Storage接口支持多种存储后端
2. **API实现扩展**: 通过Service接口可轻松添加新的API实现
3. **命令扩展**: 基于cobra的命令系统支持添加新命令
4. **配置扩展**: ConfigManager支持添加新的配置项

# 数据隔离分析报告

## 📋 分析概要

当前系统在数据隔离方面存在**严重安全漏洞**，不适合作为SaaS服务部署。所有用户的数据都混合在一起，任何用户都能访问其他用户的敏感数据。

## 🚨 关键安全问题

### 1. 请求记录数据泄露
**风险等级**: 🔴 极高

**问题描述**:
- 所有用户的API请求记录存储在同一个数据库/文件系统中
- 没有用户权限过滤，用户A能看到用户B的所有API请求和响应
- 包括敏感信息：API密钥、请求内容、响应数据

**受影响文件**:
- `pkg/api/requests_handlers.go` - GetRequests(), GetRequestByID()
- `pkg/storage/filesystem.go` - 所有存储方法
- `pkg/storage/sqlite.go` - 所有存储方法

**示例泄露数据**:
```json
{
  "id": "req-123",
  "headers": {
    "Authorization": "Bearer sk-user-a-secret-key"
  },
  "body": {
    "messages": [{"role": "user", "content": "机密商业信息"}]
  }
}
```

### 2. 代理配置数据混合
**风险等级**: 🔴 极高

**问题描述**:
- 所有用户共享同一个代理配置文件
- 用户A修改代理设置会影响用户B的服务
- API密钥和认证信息会被覆盖

**受影响文件**:
- `pkg/api/ui_handlers.go` - GetProxyConfig(), SaveProxyConfig()

### 3. 存储配置全局共享
**风险等级**: 🟡 中等

**问题描述**:
- 所有用户数据存储在同一路径
- 存储路径配置是全局的

### 4. 用户身份未在数据访问中验证
**风险等级**: 🔴 极高

**问题描述**:
- 虽然有用户认证，但在数据访问时不验证用户身份
- 获取到token后，可以访问任何用户的数据

## 🛠️ 解决方案设计

### 方案1: 用户维度数据隔离 (推荐)

#### 1.1 数据模型改造
```go
// 为Request模型添加用户字段
type Request struct {
    ID        string    `json:"id"`
    UserID    string    `json:"user_id"`    // 新增用户ID字段
    Method    string    `json:"method"`
    Path      string    `json:"path"`
    // ... 其他字段
}

// 为代理配置添加用户字段
type ProxyConfig struct {
    UserID    string    `json:"user_id"`    // 新增用户ID字段
    Enabled   bool      `json:"enabled"`
    TargetURL string    `json:"target_url"`
    // ... 其他字段
}
```

#### 1.2 存储层改造
```go
// 修改存储接口，所有方法都需要userID参数
type Storage interface {
    SaveRequest(userID string, req *Request) error
    GetRequestsByUser(userID string, limit, offset int) ([]*Request, error)
    GetRequestByUser(userID, requestID string) (*Request, error)
    DeleteRequestByUser(userID, requestID string) error
    DeleteAllRequestsByUser(userID string) error
}
```

#### 1.3 API层改造
```go
// 从token中提取用户信息
func (s *UIServer) getCurrentUser(c *gin.Context) (string, string, error) {
    token := extractToken(c)
    userInfo, found := s.tokenStore.Load(token)
    if !found {
        return "", "", errors.New("用户未认证")
    }
    
    info := userInfo.(map[string]interface{})
    username := info["username"].(string)
    userType := info["userType"].(string)
    
    return username, userType, nil
}

// 修改请求处理方法
func (s *UIServer) GetRequests(c *gin.Context) {
    userID, userType, err := s.getCurrentUser(c)
    if err != nil {
        c.JSON(401, StandardResponse{Code: 401, Msg: "认证失败"})
        return
    }
    
    // 根据用户类型决定数据访问范围
    var requests []*Request
    if userType == "root" {
        // root用户可以看到所有数据
        requests, err = s.storage.GetAllRequests(limit, offset)
    } else {
        // 普通用户只能看到自己的数据
        requests, err = s.storage.GetRequestsByUser(userID, limit, offset)
    }
    
    // ... 返回结果
}
```

### 方案2: 多租户架构 (企业级)

#### 2.1 租户隔离
```go
type Tenant struct {
    ID          string    `json:"id"`
    Name        string    `json:"name"`
    StoragePath string    `json:"storage_path"`
    CreatedAt   time.Time `json:"created_at"`
}

type User struct {
    ID       string `json:"id"`
    Username string `json:"username"`
    TenantID string `json:"tenant_id"`  // 关联租户
    Role     string `json:"role"`       // admin, user
}
```

#### 2.2 存储隔离
```go
// 每个租户使用独立的数据库/文件夹
func NewTenantStorage(tenantID string) (Storage, error) {
    tenantDataDir := filepath.Join("data", "tenants", tenantID)
    return storage.NewFileSystemStorage(tenantDataDir)
}
```

## 💡 建议

1. **立即停止SaaS部署** - 当前版本存在严重安全漏洞
2. **优先实施方案1** - 最快解决核心问题
3. **考虑方案2** - 如果需要支持企业客户
4. **添加安全测试** - 确保数据隔离有效性

## 📝 验证清单

修复完成后，需要验证以下内容：

- [ ] 用户A无法访问用户B的请求记录
- [ ] 用户A无法修改用户B的代理配置  
- [ ] root用户可以访问所有数据（如需要）
- [ ] 数据存储在用户特定的路径
- [ ] 配置文件按用户隔离
- [ ] 删除操作只影响当前用户的数据
- [ ] 导出功能只导出当前用户的数据

---

**结论**: 当前系统**不适合作为SaaS服务**，需要立即进行数据隔离改造。建议按照上述方案进行紧急修复，确保用户数据安全。
